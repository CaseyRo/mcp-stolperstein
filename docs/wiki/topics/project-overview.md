# Project Overview

## Summary [coverage: high — 8 sources]

`mcp-stolperfalle` is a FastMCP server that captures and recalls experiential knowledge — "Knowledge Units" (KUs) — for AI coding agents. A KU is a problem→action pair carrying confidence, severity, provenance, and org ownership. Agents `query()`, `propose()`, `confirm()`, `flag()`, and `reflect()` against it; the point is that Claude Code sessions build shared muscle memory across session boundaries, so an agent avoids pitfalls it has never personally seen in a given repo because a past session (in any project) captured them.

It is a **conforming-plus-extending implementation of Mozilla AI's [cq](https://github.com/mozilla-ai/cq)** standard, not a fork. On the wire it conforms strictly to the upstream `cq` schema (vendored at commit `92b35de`); locally it carries a richer superset (severity, a status state machine, a `kind` enum, DID-based provenance, multi-tenant `owner_org`). Two serializers enforce that boundary: `to_cq_json_strict()` emits upstream-valid payloads with extensions stripped, and `to_cq_json_rich()` emits the full internal shape. A third, `to_cq_v0()`, emits the legacy shape during the Siyuan-sync transition.

Strategically, the project is **Phase 1 of the "machine-readable org layer"** CDiT is building — the thesis that agent-to-agent awareness can replace the middle-management-as-compression-layer that status meetings, reports, and dashboards exist to compensate for. Multi-tenant primitives (`owner_org`, `TRUSTED_ORGS` visibility) and emergent-signal detection are foundations, not the finished product.

The codebase has passed through two large change proposals. **`stolperstein-mvp-scaffold`** built the greenfield server: six MCP tools, SQLite + FTS5 + sqlite-vec hybrid search, the confidence algorithm, embeddings, Docker/Komodo deployment, and the initial Siyuan + CQ-team sync. **`cq-v1-alignment-and-hooks`** then reframed the project as a reference implementation: it fixed strict CQ conformance, introduced a first-class migration framework (currently schema_version 6), split the Ed25519 signing key out of the DB, wired three real Claude Code hooks, and added emergent-signal detection plus org-boundary primitives. Production runs via Komodo on server `nebula-1`, exposed at `https://mcp-stolperfalle.cdit-dev.de` through a Cloudflare tunnel. Day-to-day the operator's only recurring intentional act is `/stolperfalle:reflect` at the end of a substantive session; everything else (hooks firing, query/confirm/flag calls) is agent-driven.

## Rationale & Context [coverage: high — 5 sources]

**The original problem (mvp-scaffold).** AI coding agents rediscover the same failures independently across sessions. Hard-won solutions to obscure gotchas — dependency quirks, platform restrictions, config traps — evaporate when a session ends. For a solo dev running a broad stack, the same agent hits the same wall weeks later; there is no mechanism for agents to learn from past experience across session boundaries. Rather than build a parallel system, the decision was to build on Mozilla AI's `cq` (released March 2026) — an open standard for shared agent learning with tiered storage (local/team/global commons), confidence scoring, and a defined KU JSON interchange format. Stolperfalle would be a **CQ-compatible local node** that both consumes from and contributes to the CQ ecosystem while adding local value (Siyuan sync, Claude Code hooks, richer UX).

**What forced the second change (cq-v1-alignment-and-hooks).** Re-checking upstream `cq` surfaced two surprises. First, the current upstream schema is much smaller than Mozilla's blog copy implied — no `severity`, no DID-based provenance, no `graduation_history`, no `kind` enum, no state machine; just insight + context + confidence + a flag log. The richer concepts were aspirational, not shipped. Second, Stolperfalle's own v0 implementation was already ahead of upstream in several ways but named things slightly wrong (`domain` vs upstream `domains`, 24-hex IDs vs strict 32, string `version` vs integer, top-level `last_confirmed` vs inside `evidence`). This reframed the narrative: not "catching up to CQ v1" but "conforming to its strict small base and deliberately extending it." Extensions the project finds useful get proposed upstream so the protocol grows.

**Three operational reasons forced this into one atomic change** rather than iterative cleanup:
1. **Strict conformance was broken** — `domain/domains`, ID format, version type, and `last_confirmed` location all failed upstream validation; any consumer validating the wire output would reject it.
2. **No migration path existed** — `store._init_db` was CREATE-IF-NOT-EXISTS-only, so existing KUs in the production volume would silently diverge on any schema change. The first breaking migration is now.
3. **Hooks were metadata-only** — `hooks.json` was aspirational; nothing fired, so agents kept rediscovering failures the KB could have warned them about, undermining the thesis.

A **four-agent review** (ux-auditor, mcp-tool-reviewer, security-auditor, sharp-business-strategist) surfaced fixes cheap to fold in and catastrophic to ship past (private-key separation, hook injection sanitization, honest tool annotations, Mcp-shaped errors) — folded in atomically.

**Strategic reframe.** Casey moved the project from "personal knowledge base" to "atomic data layer for the machine-readable org layer that replaces status meetings" (tracked as CDI-999). This is now Phase 1 of that larger ambition, not a personal-tool polish pass. Stakeholders: Casey (solo operator + consulting owner), future Mittelstand consulting prospects who will see this as the reference implementation of the positioning, downstream Siyuan sync, future CQ team/global-tier consumers, and future org-scale deployments where KUs cross department boundaries. An accepted risk is noted explicitly: if the CDiT positioning doesn't validate, the technical foundation (CQ v1, migrations, hooks, security) is load-bearing regardless.

## Requirements & Behavior [coverage: high — 6 sources]

**Six MCP tools**, a thin layer over `KnowledgeStore`: `query`, `propose`, `confirm`, `flag`, `reflect`, `status` — aligned with CQ's tool surface.

- **`propose()`** — takes flat `context_languages[]` / `context_frameworks[]` / `context_environment` / `context_pattern` params plus optional `severity` (default `medium`). LLMs handle flat params significantly better than nested objects, so the MCP surface is flat even though storage/wire are nested. Generates a 32-hex `ku_id`, stamps `provenance.proposer_did` + `owner_org`, initial confidence 0.5, status `draft`. Rejects `kind="gap-signal"` with an actionable `McpError` recovery hint (tool gaps are detected automatically, not proposed).
- **`query()`** — hybrid search (FTS5 `bm25()` + sqlite-vec cosine). Applies the `TRUSTED_ORGS` visibility filter BEFORE confidence/severity ranking. Severity is a ranking tiebreaker only — there is deliberately no `severity_min` filter (callers don't know what values exist in the store). Zero-result queries record a row in `query_misses` for emergent detection.
- **`confirm()`** — increments confirmations, writes `last_confirmed_at`, recalculates confidence, handles draft→active. Never touches `graduation_history`. Honestly annotated `idempotentHint=False` (it mutates counters).
- **`flag()`** — updates status (disputed/archived), records reason + detail, caps confidence at 0.5 for disputed. `reason=superseded` writes the top-level `superseded_by` column (matching upstream), not `related[]`. Also `idempotentHint=False`.
- **`reflect()`** — LLM-driven extraction (OpenAI-compatible endpoint) with a heuristic NLP fallback. Returns *candidates* pre-filled with flat `context_*` + `severity` + `kind`, never committed KUs — an agent or human must explicitly `propose()` each keeper. This is the one human-in-the-loop step.
- **`status()`** — default output is token-scoped (`total`, `by_status`, `confidence_distribution`, `staleness`, `tool_gap_signals` grandfathered-vs-emergent). `schema_version`, `proposer_did`, migration history, hook state, and `query_misses` stats move behind `status(debug=True)`.

**KU lifecycle.** State machine `draft → active → stale → disputed → archived`. Confidence scoring is diversity-weighted (distinct `owner_org` count matters more than raw confirmations), with temporal decay (linear ~0.01/day past a staleness threshold, floor 0.1) and a dispute penalty (cap 0.5). Severity `critical` raises the decay floor to 0.2; severity `low` decays to a 0.1 floor. Severity table: `low` (cosmetic), `medium` (default standard pitfall), `high` (broken build / local dev blocker, ranked above equal-confidence mediums), `critical` (breaks production if ignored).

**Claude Code plugin behavior** (the operator-facing runbook, from USING.md). Hooks fire in the background, silently, on **structured** error signals only — exception class names, tracebacks, non-zero exit codes, HTTP status strings, explicit `fatal:`/`panic:`/`Error:` prefixes — never bare conversational English like lowercase `error`/`failed`. Three hooks: `UserPromptSubmit` (injects a one-line hint on confidence ≥ 0.5 top-1 match), `PostToolUse` (matcher Bash; fires on non-zero exit or structured stderr, never delays command output), and `Stop` (nudges `/stolperfalle:reflect` only when the session had ≥20 tool turns AND ≥1 failing bash OR ≥1 `confirm`/`flag`). Rate-limited: 30s cooldown per hook type, 5min per-KU dedupe. Per-project overrides via `.claude/settings.json` (`STOLPERFALLE_HOOKS_DISABLED`, `STOLPERFALLE_REFLECT_THRESHOLD`, `STOLPERFALLE_HOOK_COOLDOWN_S`, `STOLPERFALLE_ERROR_PATTERNS`). Hooks require the server reachable over HTTP (`MCP_STOLPERFALLE_PUBLIC_URL` + `MCP_STOLPERFALLE_API_KEY`); without those, handlers exit 0 silently.

**CLI subcommands.** `mcp-stolperfalle` (run server), `migrate` (apply pending migrations + exit), `prune-backups [--confirm]` (clean `.bak-pre-v<N>` files), `detect-emergent` (run emergent aggregation manually).

## Design & Architecture [coverage: high — 6 sources]

**Language + framework.** Python 3.12+ with FastMCP — chosen because FastMCP is the most mature MCP server framework, `cq` itself is Python, and sqlite-vec + sentence-transformers have first-class Python support. TypeScript and Rust were rejected (embedding-ecosystem friction; overkill for a solo-operator tool).

**Storage.** A single SQLite file with FTS5 and sqlite-vec, chosen over Postgres+pgvector for zero operational overhead (no separate service, single-file backup, trivial Docker volume mount; WAL mode handles concurrent reads during writes). `knowledge_units` carries **flat columns** (one per scalar; JSON only for actual arrays/objects like `context_frameworks`, `contributing_orgs`, `related`, `graduation_history`). Pydantic presents the nested CQ shape (`Context`, `Evidence`, `Provenance`) in-memory and on-wire. Full normalization was rejected (4× join cost on every read against the FTS5/vec0 rowid linkage); a single `payload JSON` blob was rejected (breaks indexable `confidence`/`status`/`severity`).

**Embeddings.** `all-MiniLM-L6-v2` (384-dim) via sentence-transformers in-process (~80MB, CPU-fast, offline, no API cost), with an optional HTTP fallback via `CQ_EMBEDDING_API_URL`. FTS5 is the primary search path; vector search is additive, so a sqlite-vec failure degrades to keyword-only with a logged warning. Embedding input concatenates summary + detail + action, extended to include `context.pattern` when present (other `context_*` fields are too sparse and would dilute signal).

**Migrations.** An in-repo framework, not Alembic (whose ORM bias fights raw `sqlite3` + `sqlite_vec.load`; expected lifetime migration count is small and linear, so a ~50-line runner wins). `store._init_baseline()` creates the v0-shape tables on a fresh install; then `migrations.run()` applies ordered `mNNNN_*.py` modules to bring the DB to current (schema_version 6). Each migration declares `version: int`, `breaking: bool`, `slug: str`, `up(conn)`. The runner runs at the head of `_get_db()`, one transaction per migration (raises roll back automatically). Before any `breaking=True` migration it copies the DB to `<db>.bak-pre-v<N>` and **refuses to overwrite an existing backup** (forces operator intervention rather than silently clobbering a prior aborted migration). The migration chain: `m0001` (pad IDs to 32 hex + rewrite refs), `m0002` (domain→domains, `last_confirmed_at`, `superseded_by` hoist, `context_languages/frameworks/pattern`), `m0003` (Stolperfalle extensions `evidence_severity`, `context_environment`), `m0004` (`install_identity` table + private-key file split + `proposer_did`/`graduation_history`/`owner_org`), `m0005` (gap-signal → tool-gap-signal grandfathering, additive), `m0006` (`query_misses` rolling table, additive).

**Provenance + identity.** One Ed25519 keypair per install (via `cryptography`), identifier `did:key:z6Mk...` derived by multicodec + multibase on the public key, never auto-rotated. **The private key lives OUTSIDE the DB** — `/data/stolperfalle.key` (mode 0o600) or the `MCP_STOLPERFALLE_SIGNING_KEY` env var (base64, takes priority). The DB holds only the public key + DID string. This is security-critical (security-review H1): Docker volume backups, `docker cp`, and misconfigured mounts all expose the data DB, and co-locating the signing key would let a DB leak forge `proposer_did` attribution, destroying the audit trail the DID exists to provide. `graduation_history[]` entries carry an `agent: true` marker so automated graduation is distinguishable from human-initiated. `did:web` and per-session DIDs were rejected (hosting requirement; diversity-math inflation). Full W3C DID-document resolution is a non-goal — `did:key:z...` is an opaque identifier, no registry, no resolver.

**Emergent signals** (`emergent.py`). Simple count-based clustering over `query_misses`: cosine ≥ 0.8, ≥5 misses across ≥2 distinct sessions/hour-buckets emits a new `tool-gap-signal` KU marked `provenance.emergent=true`, with 7-day dedupe against recent emergent KUs. Triggered fire-and-forget on every Nth `query()` (`EMERGENT_DETECT_EVERY_N`, default 10), or via the CLI. Disableable via `STOLPERFALLE_EMERGENT_DISABLED` or `EMERGENT_DETECT_EVERY_N=0`. Deliberately not ML-sophisticated — the point is the capability EXISTS as a real module so later changes can swap the algorithm without schema/tool churn.

**Hooks** (`plugin/stolperfalle/hooks/handlers/`). Python handlers, stdlib-only (Claude Code runs them with whatever `python3` is on PATH; no fastmcp import), POSTing to `/hook/*` REST endpoints with a ~500ms/1.5s budget. Injection is sanitized: `action_sanitized = re.sub(r'<[^>]+>', '', action)` strips angle brackets (security M1), a fixed wrapper template is used, and content is never wrapped in `<system-reminder>`-shaped tags the host model might elevate. Rate-limit/dedupe state is an `fcntl.flock`-guarded JSON file at `$FASTMCP_HOME/hooks-state.json`, schema-validated on read, reset on corruption. Bearer-token hygiene (H2): errors are re-raised without the `Authorization` header, the `_token` var is `del`-eted after the request, never passed to `subprocess(env=...)`. Same sanitization runs at team-sync ingest (M3).

**Auth** (`auth.py`). FastMCP `MultiAuth` — Cloudflare Access / Keycloak OIDC for browser OAuth clients plus static bearer tokens (`stmcp_` prefix, `hmac.compare_digest`) for Claude Code / n8n. Engages only when `TRANSPORT=http`; stdio has zero auth overhead. Auto-generates an `stmcp_` key with a warning if unset.

**Sync** (`sync/`). `cq_team.py` emits strict-CQ on graduation and validates inbound payloads against the vendored schema before sanitizing + storing. `siyuan.py` honors `CQ_SIYUAN_SCHEMA_VERSION` (0 = legacy shape during transition), an async fire-and-forget queue with exponential-backoff retry (max 3) so sync never blocks a tool response. Both are env-gated and fully optional.

**Deployment.** Docker (python:3.12-slim multi-stage, embedding model downloaded at build time, non-root `mcp` user, healthcheck on `/mcp`, port 8716). Production is Komodo on `nebula-1` (auto-deploys from `main`), fronted by a co-located Cloudflare tunnel; SQLite persists to volume `stolperfalle-data`, FastMCP OAuth cache to `fastmcp-data`. Config is all env vars via Pydantic `BaseSettings` (`SecretStr` for sensitive values) — no config file, Komodo-compatible.

## Schema & Interop [coverage: high — 5 sources]

**Conformance posture: conform strictly on the wire, extend richly locally.** The wire representation used for CQ team/global sync and any external validation conforms exactly to the upstream `knowledge_unit.json` schema (vendored at SHA `92b35de` under `tests/fixtures/cq/`, with `CQ_SCHEMA_REF.md` recording the pin; `jsonschema` in the dev group asserts every strict serialization validates). Because upstream is `additionalProperties: false`, extensions cannot ride along in strict output — they are stripped at the serializer boundary.

**Strict-CQ mapping** (`to_cq_json_strict()`): `domain` → `domains` (plural), `ku_id` matches `^ku_[0-9a-f]{32}$`, `version` is the integer `1` (not a semver string), `context.languages` is an array, `evidence.last_confirmed` sits inside the evidence block, `created_by` is derived from `proposer_did`, and flag reasons map to upstream's enum (`stale | incorrect | duplicate`) — the local `dangerous` maps to `incorrect` + a local marker, and `superseded` is never a flag (it's the top-level `superseded_by` field). Inbound KUs from a CQ team API must validate against the vendored schema before ingest, then get `summary`/`detail`/`action` sanitized + length-capped (closing a prompt-injection vector via team sync).

**Three serializers.** `to_cq_json_strict()` (upstream-valid, extensions stripped), `to_cq_json_rich()` (full internal superset for local dumps/debugging/consumers that want the extensions), `to_cq_v0()` (pre-change legacy shape, gated by `CQ_SIYUAN_SCHEMA_VERSION=0`, explicitly temporary until Siyuan sync is updated).

**Stolperfalle extensions** — all local-only, all documented in `docs/cq-extensions.md`: `evidence.severity` (ranking tiebreaker + decay floor), `evidence.contributing_orgs`, `context.environment` (build/runtime version scoping), top-level `kind` (`pitfall | workaround | tool-recommendation`), `status` state machine, `staleness_policy`, `related[]` relationship graph, `owner_org`, and rich `provenance` (`proposer_did`, `graduation_history`, `emergent`). `tool-gap-signal` is produced only by aggregation, never by `propose()`.

**Positioning as a reference implementation.** The project is framed publicly as a conforming-plus-extending reference implementation, not a fork — extensions found useful get proposed upstream so the protocol grows. The first batch was filed as a discussion on `mozilla-ai/cq`: [mozilla-ai/cq#286](https://github.com/mozilla-ai/cq/discussions/286). This visible relationship between the code and the protocol is part of the CDiT positioning.

## Status & Open Questions [coverage: high — 6 sources]

**Shipped (both changes largely complete).** The MVP scaffold delivered all six tools, hybrid store, confidence algorithm, embeddings, Docker/Komodo deploy, Caddy/Cloudflare ingress, Keycloak client, and CQ-team + Siyuan sync. The cq-v1 change delivered strict conformance, the migration framework (schema_version 6), DID provenance with key separation, three live hooks, emergent detection, org-boundary primitives, the rewritten SKILL.md, and the upstream discussion (#286 filed, linked into CDI-999). Most tasks across both `tasks.md` files are checked.

**Explicitly deferred to Phase 2** (foundation-only in Phase 1):
- **Enforceable org boundaries.** `owner_org` is recorded and `TRUSTED_ORGS` is read-filter-only with default `*` (trust-all, preserving single-install behavior). Write-side permissions, per-org read/propose/graduate rights, inheritance, and per-org UI are Phase 2. Adding the column now is a cheap migration; retrofitting later means rewriting every KU's visibility.
- **Richer emergent aggregation.** Phase 1 is conservative count-based miss-clustering; ML clustering / LLM summarization of clusters is Phase 2.
- **Graduation review UI.** `graduation_history` is recorded passively; an explicit human-facing review/graduation UI and selective cross-org graduation are Phase 2.
- **Upstream adoption** of the extensions tracked in #286.

**Non-goals** (won't be built here): full W3C DID resolution; re-inferring `context_*` from existing summaries (migration sets them NULL, filled on later touch); reworking Siyuan/CQ-team HTTP semantics (only payload shape changed); and billing / tenancy-provisioning / org-onboarding UX — the positioning implies a consulting engagement, not a SaaS.

**Open questions from the design docs.** From cq-v1: emergent-aggregation cadence (event-based every-Nth-query vs time-based hourly — current draft event-based, `EMERGENT_DETECT_EVERY_N=10`); whether team-sync import should append a `graduation_history` entry (current draft: no, imports stay passive); hook-handler packaging (Python stdlib-only, chosen, documented PATH requirement); the Phase 2 permissions model shape (ACL-per-org vs role-based vs capability-based — deferred); and the CDiT consulting engagement shape (6- vs 12-week, fixed-price vs T&M, first-client profile — CDI-999, not a technical question but shapes Phase 2's definition of "good"). From mvp-scaffold: embedding dimension (384 vs 768); hook granularity (fire on all tool errors vs Bash/Read only — since narrowed to structured signals); CQ team-API auth (API keys vs also OIDC); and the KU-dedup cosine threshold (0.9 proposed, needs validation).

**Known operational caveats.** Default-trust `TRUSTED_ORGS='*'` becomes dangerous at multi-tenant scale — Phase 2 is required before any multi-org deployment. The private signing key file is sensitive and must be excluded from volume backups and `docker cp`; if lost, the install generates a new keypair on next boot (KU content intact, but the DID rotates and the audit chain visibly breaks at that point). Note the deploy docs still carry some staleness: `komodo.toml` references old `KEYCLOAK_*` vars (source of truth for new deployments is `compose.yaml` + `.env.example`), and the former host `ubuntu-smurf-mirror` named in the mvp tasks and cq-v1 migration plan is EOL and removed from Komodo — current prod is `nebula-1`.
